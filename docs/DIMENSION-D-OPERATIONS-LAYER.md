# 维度 D：运营层改造方案

**改造目标**：AI 能力可度量、可配置、可进化。回答三个核心问题：AI 有没有用？AI 出错了怎么办？如何让 AI 越来越好？

---

## D1. AI 效果度量：A/B 框架

### 核心问题

8 个 AI 引擎部署后，没有任何指标回答"启用 AI 后招聘效果有没有变好"。

### 方案

#### D1.1 度量框架

每个触点独立度量，不混在一起：

| 触点 | 核心度量指标 | 数据来源 | 对比方式 |
|------|-------------|----------|----------|
| 1 需求诊断 | 需求→Offer 周期缩短率 | demand + offer 时间戳 | 启用 AI 诊断的需求 vs 直接填表单的需求 |
| 2 面试辅助 | 面试评价一致性（Kappa） | interview 评分 | 使用 AI 辅助的面试 vs 不使用 |
| 3 校准会 | 校准后评分修正率 | calibration 记录 | 有 AI 主持的校准会 vs 无 AI |
| 4 意向预测 | Offer 接受率 | offer 结果 | 有意向预测的候选人 vs 没有 |
| 5 周期预测 | 预测误差（实际 vs 预测） | 预测表 + 实际到岗日 | 持续追踪 |
| 6 Offer 策略 | Offer 接受率 + 薪酬成本 | offer 结果 | 使用 AI 策略的 Offer vs 没有 |
| 7 面试官质量 | 面试官评分预测准确度提升 | interviewer profile | 有 coaching 的面试官 vs 没有 |
| 8 人才密度 | 新员工 90 天绩效达标率 | onboard 绩效 | 按 AI 建议招聘的 vs 自行招聘的 |

#### D1.2 A/B 实验设计

对于可以对照的触点（如意向预测、Offer 策略），采用标准 A/B：

```
同一岗位族的在招岗位随机分为两组：
  A 组（实验组）：启用 AI 触点
  B 组（对照组）：不启用

每组至少 20 个岗位 × 每个岗位至少 5 个候选人

运行 4-8 周
→ 统计检验（t 检验 / Mann-Whitney U）
→ 如果 p < 0.05 且效应量 > 最小可检测效应量
→ 确认 AI 有效
→ 全量推送
```

#### D1.3 度量 Dashboard

在现有 `Dashboard.vue` 新增 **AI 价值** 模块：

```
┌────────────────────────────────────────────┐
│ AI 价值概览                                  │
├──────────┬──────────┬──────────┬───────────┤
│ 意向预测  │ Offer策略 │ 需求诊断  │ 面试辅助   │
│ 提升接受率 │ 降低薪酬  │ 缩短周期  │ 提升一致性  │
│  +12%    │  -8%     │  -5天    │  +15%     │
│  ✅ 有效  │  ✅ 有效  │  ⏳ 进行中 │  ✅ 有效   │
├──────────┴──────────┴──────────┴───────────┤
│ 周期预测 MAE：3.2 天 （目标 < 5 天）         │
│ 人才密度：本月 0.72 → 上月 0.68（↑ 6%）     │
└────────────────────────────────────────────┘
```

## D2. 审计日志：AI 决策全追溯

### 需求

任何 AI 建议被采纳/忽略/覆盖，必须完整记录。支持：
- 合规审计（"为什么这个候选人被 AI 推荐？"）
- 模型改进（"为什么这个建议被忽略了？"）
- 责任追溯（"谁做了最终决定？"）

### 数据表

`brain_ai_decision_log`（已在维度 A 中定义）。

### 记录时机

| 事件 | 记录内容 |
|------|----------|
| AI 生成建议 | touchpoint + target + ai_output_json + confidence |
| 用户采纳建议 | user_action = ACCEPTED + user_id |
| 用户忽略建议 | user_action = IGNORED + user_id（可选填原因） |
| 用户修改 AI 建议 | user_action = OVERRIDDEN + override_reason + 修改前后对比 |
| 业务结果产生 | business_outcome（如 OFFER_ACCEPTED / PROBATION_PASSED） |

### 审计查询支持

- "过去 30 天，面试辅助建议被采纳的比例是多少？"
- "某面试官忽略了 AI 的偏差提醒，后续他的评分准确度变化？"
- "意向预测建议'放弃'的候选人中，有多少实际上接了其他公司的 Offer？"

## D3. 反馈闭环：让 AI 从人的决策中学习

### 核心问题

当前 AI 给建议 → 人决定 → 结束。AI 不学习"为什么人被忽略"。

### 方案

#### D3.1 忽略原因收集

当用户忽略 AI 建议时，轻量级弹窗：

```
┌────────────────────────────────────────────┐
│ 你忽略了 AI 的建议：「安排 Hiring Manager 沟通」 │
│                                             │
│ 原因（可选）：                                │
│ ○ 画像不匹配 — 候选人实际不符合该岗位          │
│ ○ 时机不合适 — 候选人已进入终面，不宜打扰      │
│ ○ AI 判断不准 — 我认为候选人意向没有问题       │
│ ○ 其他 ______                               │
│                                             │
│ [提交] [跳过]                                │
└────────────────────────────────────────────┘
```

#### D3.2 业务结果回流

AI 建议被采纳后，追踪最终业务结果：

```
AI 建议：录用候选人（意向 85 分）
HR 采纳 → 发 Offer → 候选人接受 → 入职
  → 90 天后绩效评估：达标 ✅
  → 回流：该意向预测模型的这个判断是正确的

AI 建议：放弃候选人（意向 25 分 + 薪资差距 40%）
HR 忽略 → 发 Offer → 候选人拒绝
  → 回流：AI 判断正确，HR 忽略建议导致了流程浪费
```

这些反馈用于：
1. 模型重训练时的样本权重调整
2. 用户层面的"AI 建议质量"指标

#### D3.3 连续学习管道

```
每周定时任务：
  1. 从 brain_ai_decision_log 中提取本周有业务结果的记录
  2. 对照 AI 建议 vs 实际结果
  3. 计算各触点的 Precision / Recall / F1
  4. 如果某个触点指标连续 2 周下降：
     - 触发告警（通知管理员）
     - 检查是否需要模型重训练
  5. 如果累积 100+ 新样本：
     - 触发模型自动重训练（维度 B1.6）
```

## D4. 可配置性：租户级 AI 设置

### 需求

不同企业对 AI 的接受度不同。激进型公司希望 AI 更多介入，保守型公司希望 AI 只在后台提供参考。

### 管理后台设计

在 `Settings` 模块新增 **AI 设置** 页面：

```
┌────────────────────────────────────────────┐
│ AI 设置                                     │
├────────────────────────────────────────────┤
│ 全局模式：                                   │
│ ○ 激进  — AI 主动推送建议，允许高置信度自动执行  │
│ ● 标准  — AI 主动推送建议，所有操作需人工确认    │
│ ○ 保守  — AI 被动响应查询，仅提供参考信息       │
├────────────────────────────────────────────┤
│ 按触点开关：                                 │
│ ✅ 需求诊断   激进程度：[──●──] 标准          │
│ ✅ 面试辅助   激进程度：[─●───] 保守          │
│ ✅ 校准会     激进程度：[──●──] 标准          │
│ ✅ 意向预测   激进程度：[───●─] 激进          │
│ ⬜ 周期预测   (未启用)                       │
│ ✅ Offer策略  激进程度：[──●──] 标准          │
│ ✅ 面试官质量 激进程度：[─●───] 保守          │
│ ✅ 人才密度   激进程度：[──●──] 标准          │
├────────────────────────────────────────────┤
│ 高级设置：                                   │
│ 意向评分阈值：HIGH > [70] MEDIUM > [40]     │
│ 面试官偏差告警：宽松指数偏离 > [0.25]         │
│ LLM 模型选择：[GPT-4o ▼]                    │
│                                             │
│ [保存设置] [恢复默认]                         │
└────────────────────────────────────────────┘
```

**激进程度的含义**：

| 模式 | AI 建议呈现 | 操作确认 | 自动执行 |
|------|------------|----------|----------|
| 激进 | 主动弹窗 + 卡片高亮 | 关键操作需确认 | 低风险操作可自动执行 |
| 标准 | 卡片展示 + 角标提示 | 所有操作需确认 | 无自动执行 |
| 保守 | 被动查询才展示 | 所有操作需确认 + 二次确认 | 无自动执行 |

所有配置变更记录到 `brain_ai_decision_log`（touchpoint = CONFIG_CHANGE）。

## D5. 冷启动运营

### 新客户上线流程

```
Day 0：客户注册
  → 系统自动选择行业模板（根据客户填写的行业 + 规模）
  → AI 设置默认为「标准」模式

Day 1-30：数据积累期
  → 所有 AI 引擎运行在行业模板上
  → 每个 AI 输出明确标注：「基于行业数据，置信度较低」
  → 收集：Offer 结果、面试评价、入职绩效

Day 30-90：过渡期
  → 累积 50+ 样本的触点开始切换为租户微调模型
  → 标注更新为：「正在学习贵司模式，置信度提升中」

Day 90+：成熟期
  → 累积 200+ 样本的触点完全切换为租户专属模型
  → 置信度最高
  → AI 效果 Dashboard 开始有统计显著的结果
```

## D6. 运营层改造验收标准

- [ ] A/B 实验框架就绪，支持按岗位族随机分流
- [ ] Dashboard 展示 AI 价值指标（至少 3 个触点的效果数据）
- [ ] `brain_ai_decision_log` 表在每次 AI 建议生成/采纳/忽略时写入
- [ ] 忽略 AI 建议时弹出原因收集对话框
- [ ] 业务结果回流管道就绪（Offer 结果 → 意向模型反馈）
- [ ] 管理后台 AI 设置页面（全局模式 + 触点开关 + 阈值配置）
- [ ] 行业模板就绪（至少 3 个行业 × 冷启动模型）
- [ ] 模型自动重训练触发器就绪
