# 维度 C：嵌入层改造方案

**改造目标**：AI 从"独立页面"嵌入到"用户自然工作流"，渐进式披露，每个输出附带可点击动作。

**设计原则（来自 PRD v7）**：
- 原则 6：Insight → Action — 每条 AI 洞察通向一个可执行操作
- 原则 7：Embedded, Not Separate — AI 在用户自然工作流中出现

---

## C1. 嵌入策略：从独立页面到工作流内组件

### v6 现状（反模式）

```
用户路径：
  面试官在 InterviewBoard Kanban 看到候选人
  → 点击候选人
  → 想获取 AI 面试辅助
  → 必须导航到 /ai/interview/{id}/scorecard （独立页面）
  → 看完 AI 建议
  → 导航回 Kanban
  → 开始面试
```

AI 没有出现在面试官**正在做事的页面**上。

### v7 目标

```
用户路径：
  面试官在 InterviewBoard Kanban 看到候选人
  → 点击候选人
  → 右侧滑出 AI 辅助面板（同一页面，不跳转）
  → 面板中显示：
    - 简历关键信号（强信号/注意点）
    - 该面试官负责维度的评分锚点
    - 建议追问方向
    - [开始面试] 按钮
  → 面试中：面板实时显示维度覆盖状态
  → 面试后：面板自动生成结构化记录草稿
```

### 嵌入映射表

| 触点 | v6 位置（独立页面） | v7 位置（嵌入） | 嵌入方式 |
|------|---------------------|-----------------|----------|
| 1 需求诊断 | `/ai/demand/create` | `DemandCreate.vue` 的第一步 | 替换"填表单"为"AI 对话" |
| 2 面试辅助 | `/ai/interview/:id/scorecard` | `InterviewBoard.vue` 的右滑面板 | 抽屉式侧边栏 |
| 3 校准会 | `/ai/calibration/:candidateId` | `InterviewEvaluation.vue` 的评价汇总页 + 独立会议页 | Tab 内嵌 + 独立路由 |
| 4 意向预测 | `/ai/intent/:candidateId` | `PipelineBoard.vue` 的候选人卡片 + `CandidateDetail.vue` | 卡片角标 + 详情 Tab |
| 5 周期预测 | `/ai/cycle-prediction/:jobId` | `Dashboard.vue` + `JobHiringSummary.vue` | Dashboard 卡片 + 岗位页段落 |
| 6 Offer 策略 | `/ai/offer-strategy/:candidateId` | `OfferList.vue` 的 Offer 创建流程 | 创建流程中的步骤 |

| 8 人才密度 | `/ai/talent-density/:orgId` | `OrgManage.vue` 的组织视图 + Dashboard | 嵌入组织架构页 |

## C2. 各触点嵌入详细设计

### C2.1 触点 2：面试辅助嵌入 Kanban

**嵌入位置**：`InterviewBoard.vue` 的面试官视图

**交互流程**：
1. 面试官在 Kanban 上看到分配给自己的候选人
2. 点击候选人卡片 → 右侧滑出 **AI 面试辅助面板**
3. 面板分三个区域：

**区域一：面试前（5 分钟准备）**

│ ⚠️ 注意：最近一份工作仅 8 个月          │
├─────────────────────────────────────┤
│ 建议追问                             │
│ • "你在 XX 产品中做的最难的取舍是什么？"  │

│ 评分锚点（Product Sense）             │

### C2.3 触点 1：需求诊断改为对话式

**当前**：`DemandCreateAI.vue` 是两步走 —— 用户填一段描述 → AI 返回诊断结果。

**改造**：改为多轮对话。

```
系统：请描述你的业务目标
用户：Q3 要上线新支付系统

系统：现有团队为什么做不到？是缺人、缺经验还是缺带宽？
用户：缺经验，现在 4 个人都没做过支付

系统：如果这个岗位不招，最大的业务风险是什么？
用户：Q3 上线会延期，可能丢客户

系统：入职第 90 天最重要的一个产出应该是什么？
用户：独立交付支付核心模块并通过压测

系统：[生成诊断报告]
     基于你的描述，你需要的不是"一个后端"，而是一个
     有支付系统经验的架构师（P8），因为：
     1. ...
     2. ...
     [确认，生成招聘方案] [调整描述] [更多追问]
```

每轮追问背后由 `DemandDiagnosisEngine` 的 LLM 驱动，但流程是对话式的，不是一次性表单。

### C2.4 触点 3：校准会改为交互式会议工具

**当前**：静态表格展示多面试官评分对比。

**改造**：在校准会页面上实现：
1. **逐步引导**：按维度逐一讨论（"我们现在讨论 Product Sense"）
2. **可修改评分**：面试官可以直接在页面修改自己的评分并填写修正原因
3. **实时统计更新**：评分修改后 Kappa、ICC 实时刷新
4. **修正历史**：所有修正自动记录到校准会记录表
5. **结论生成**：会议结束时一键生成校准报告

## C3. 渐进式披露：三层信息密度

每个 AI 输出遵循三层递进：

```
Layer 1: 一句话结论（5 秒扫读）
  「意向 82 分，高意向，1 个风险点」
  「Product Sense 存在 2 分争议，建议讨论」

Layer 2: 核心依据（30 秒理解）
  展开后的关键信号、主要影响因素、置信度

Layer 3: 完整细节（需要时深入）
  SHAP 特征贡献、历史趋势、原始数据
```

**实现方式**：
- Layer 1：卡片/角标/Tooltip
- Layer 2：侧边栏/Tab 内容
- Layer 3：详情页/展开区域

## C4. Action Button：每个洞察通向一个操作

| AI 洞察 | Action Button | 点击后行为 |
|---------|---------------|------------|
| 「意向下降 15 分」 | `[安排 Hiring Manager 沟通]` | 弹出日程选择器，创建 1:1 会议 |
| 「薪资期望高于预算 22%」 | `[评估薪酬弹性]` | 打开 Offer 策略面板 |
| 「Product Sense 评分分歧」 | `[开始讨论此维度]` | 校准会进入该维度讨论模式 |
| 「筛选通过率低」 | `[调整筛选门槛]` | 打开 OpsPack 编辑面板 |
| 「候选人沉默 5 天」 | `[发送重新激活话术]` | 打开话术选择器，预填候选人和模板 |
| 「面试官 A 需重新认证」 | `[安排认证培训]` | 弹出培训安排表单 |

所有 action 记录到 `brain_ai_decision_log`。
