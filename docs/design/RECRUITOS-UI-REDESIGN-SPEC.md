# RecruitOS UI Redesign Spec v1

## 1. 为什么要做这次重构

当前系统不是单纯的“样式不够好看”，而是缺少统一的视觉系统。

主要问题：
- 页面骨架不统一
- 信息层级不清晰
- 操作优先级混乱
- 表格、卡片、筛选区节奏不一致
- 列表、详情、工作台三类页面没有各自的视觉语言

本次重构的目标不是做炫技式界面，而是把 RecruitOS 重建为一套**稳定、克制、可扩展**的企业级招聘操作系统。

---

## 2. 设计目标

### 2.1 业务目标
- 降低招聘流程中的认知负担
- 提升 HR、面试官、管理员的操作效率
- 让系统更像“产品”，而不是“后台模板”

### 2.2 体验目标
- 页面一眼能看懂：我是谁、我在哪、下一步该做什么
- 不同页面之间保持一致的节奏和空间语言
- 状态、风险、待办事项可快速被扫描

---

## 3. 设计原则

### 原则 1：页面即任务
每个页面围绕一个核心任务展开，不做万能页面。

### 原则 2：信息先行，操作后置
先让用户理解状态，再呈现操作入口。

### 原则 3：克制视觉噪声
减少边框、阴影、渐变和强色块堆叠。

### 原则 4：全局节奏优先局部装饰
先统一字号、间距、对齐与留白，再做局部强调。

### 原则 5：状态可扫描
列表和看板必须支持“扫一眼就看懂重点”。

### 原则 6：语义复用高于样式复用
相同业务含义，使用相同视觉表达。

## 4. 页面类型体系

建议将系统所有页面收敛为 5 类标准页面。

### 4.1 List Page（列表页）
适用页面：
- CandidateList
- ResumeList
- OfferList
- DemandList
- ReferralList
- HeadhunterVendor
- UserManage
- OrgManage
- RoleManage
- TenantList

标准结构：
1. Page Header
2. Optional Stats Band
3. Filter Bar
4. Data Table
5. Pagination

核心要求：
- 标题和主操作放在同一行
- 筛选条件不超过主视觉区域的第一屏
- 表格只承担信息扫描，不承担装饰

### 4.2 Board Page（看板/泳道页）
适用页面：
- InterviewBoard
- PipelineBoard
- DemandBoard

标准结构：
1. Page Header
2. Context Switcher
3. Board Surface
4. Card Grid

核心要求：
- 列宽稳定，卡片密度适中
- 每张卡片只保留 1 个主操作
- 状态信息固定出现在统一位置

### 4.3 Workspace Page（工作台页）
适用页面：
- CandidateWorkspace
- JobDetail
- Today
- Inbox
- ChannelStaging
- AgentWorkflow

标准结构：
1. Object Header
2. Context Bar
3. Main Panel / Side Rail
4. Optional Tabs

核心要求：
- 页面必须先回答“当前对象是谁”
- 主区展示内容，侧轨展示决策与动作
- 不要把所有操作平铺到同一层级

### 4.4 Detail Page（详情页）
适用页面：
- CandidateDetail
- OfferDetail
- ResumeDetail
- TenantDetail
- DemandDetail
- JobDetail 的非工作台视图

标准结构：
1. Breadcrumb
2. Object Header
3. Key Summary
4. Section Panels

核心要求：
- 详情页不是信息堆砌，而是分层展示
- 高频字段优先，长尾字段折叠
- 返回路径必须稳定

### 4.5 Config Page（配置/设置页）
适用页面：
- TenantSetting
- SsoConfig
- LicenseInfo
- CommunicationSafety
- EvolutionWeight

标准结构：
1. Page Header
2. Section Cards
3. Inline Edit / Dialog Edit

核心要求：
- 表单不堆叠太多层级
- 保存、重置、说明文字保持一致位置
- 设置页要安静，不抢视觉焦点

## 5. 全局视觉系统（Token）

### 5.1 颜色系统
建议保留现有蓝主色，但强化中性色阶。

主色：
- Primary: 用于主按钮、选中态、关键链接、进度标识

语义色：
- Success: 通过、完成、已入职、已解析
- Warning: 待处理、暂停、即将到期
- Danger: 拒绝、失败、高风险
- Neutral: 默认标签、辅助信息

使用规则：
- 语义色不能作为大面积背景主色
- 一个页面内同色不超过 3 处强表达
- 状态标签、流程节点、数值趋势使用统一语义映射

### 5.2 字体层级
建议收敛为 4 个层级。

H1 页面标题
- 用途：页面主标题
- 语气：简短、明确

H2 区块标题
- 用途：模块、卡片区、设置分组
- 语气：解释当前区域作用

H3 小标题
- 用途：表格区标题、详情区块标题
- 语气：比正文稍强即可

正文 / 辅助文字
- 正文：主要阅读内容
- 辅助：说明、时间、备注、来源

规则：
- 不要在页面内随意新建字号
- 所有页面标题、区块标题、辅助说明统一走系统层级

### 5.3 间距系统
建议统一为：
- 4：紧凑内部间隙
- 8：组件内部常见间隙
- 12：小模块间隔
- 16：内容区标准间隔
- 24：区块间隔
- 32：页面级节拍间隔

当前问题：
- 页面中频繁出现 14、18、20、22、28 等非系统化数值
- 卡片、表格、筛选区之间缺少统一节拍

### 5.4 圆角系统
- Small：按钮、输入框、标签、小控件
- Medium：卡片、面板、表格容器、筛选栏
- Large：Dialog、Drawer、大浮层
- Full：胶囊标签、徽标

规则：
- 同一层级不要出现两种以上圆角
- 列表类页面视觉应偏“安静”，圆角不宜过大

### 5.5 阴影与层级系统
建议只保留 3 个层级表达。

Surface Level 0
- 页面背景

Surface Level 1
- 内容容器、卡片、筛选栏

Surface Level 2
- Hover、浮层、弹窗、Popover

规则：
- 默认内容容器优先使用边框，不要滥用阴影
- hover 和浮层才用轻微抬升
- Dialog / Drawer 允许明显抬升

## 6. 组件规范

### 6.1 PageHeader
统一规则：
- 左侧：标题 + 可选副标题
- 右侧：主操作区
- 主操作最多 2 个
- 返回按钮只在对象页出现，不在列表页滥用

### 6.2 FilterBar
统一规则：
- 左侧放筛选条件
- 右侧放操作按钮（搜索、重置、新增、导出）
- 筛选条件不超过一行展示
- 若超过一行，使用折叠筛选

### 6.3 StatsRow
统一规则：
- 用于列表页顶部或工作台概览
- 最多 4 张统计卡
- 不要把统计卡做太大，避免喧宾夺主
- 统计卡只表达“结果”，不表达“操作”

### 6.4 DataTable
统一规则：
- 表头中灰，行高统一
- 主字段保持蓝色链接样式
- 状态列固定使用语义 Tag
- 操作列固定右侧
- 关键字段左对齐，数值/状态居中

### 6.5 StatusTag
统一规则：
- Draft / Pending / Active / Paused / Closed / Rejected 等状态全局统一
- 一个对象同一时间只展示一种主状态标签
- 若同时存在主状态 + 子状态，主状态放前面

### 6.6 Board Card
统一规则：
- 顶部：主标识（候选人 / 面试 / 任务）
- 中部：关键属性
- 底部：状态 + 1 个主操作
- 不要在卡片里放 3 个以上文字链接

### 6.7 EmptyState
统一规则：
- 标题说明当前为什么为空
- 说明文案补上下一步建议
- 主按钮最多 1 个，辅按钮最多 1 个
- 空状态不要用太大插画，保持克制

## 7. 核心页面蓝图

## 7.1 Dashboard
当前问题：
- 信息密集但缺乏主次
- Bento 布局偏装饰性
- 今日任务不够突出

重构方案：
- 区域 A：问候 + 今日待办 + 今日面试
- 区域 B：4 个核心指标卡
- 区域 C：左侧今日任务流，右侧活跃职位 / 最近活动

设计要求：
- Dashboard 的目标是“立刻开始工作”
- 不要为了好看堆砌图表
- 行动入口优先于数据展示

## 7.2 CandidateList
当前问题：
- 操作列密度高
- 来源、解析状态、阶段等信息未统一位置
- 批量操作与筛选关系不清晰

重构方案：
- 顶部：统计 4 卡
- 筛选栏：关键词、来源、状态、日期
- 表格：姓名、公司、职位、来源、阶段、更新时间、操作
- 批量操作固定在筛选栏右侧或表格下方统一区域

## 7.3 CandidateWorkspace
当前问题：
- 页面既有详情属性，又有工作台属性
- 右侧操作轨按钮过多
- 候选人与职位之间的上下文不够清晰

重构方案：
- 对象头：候选人名 + 公司 + 职位 + 当前阶段
- 左主区：简历、评估、进展 tab
- 右侧轨：当前职位、阶段、下一步、主操作、近期进展
- 操作按优先级压缩为 3 个主要动作，其余进更多

## 7.4 CandidatePool
当前问题：
- 搜索感过强
- 职位上下文提示不够突出
- 卡片信息层级偏平

重构方案：
- 顶部强制上下文条：先选职位
- 搜索区：关键词 + 技能标签
- 结果区：卡片视图 / 列表视图切换
- 卡片统一展示姓名、公司、年限、技能、匹配结论

## 7.5 ResumeList / ResumeUpload
当前问题：
- 列表页统计与表格节奏紧
- 上传页缺少清晰的三段式反馈

重构方案：
- 列表页：标题 → 统计 → 筛选 → 表格 → 分页
- 上传页：拖拽区 → 待上传文件 → 解析结果
- 解析状态、来源、操作统一到固定列结构

## 7.6 InterviewBoard
当前问题：
- 卡片信息密度不统一
- 操作入口散落

重构方案：
- 顶部按轮次切换
- 每张卡片统一：候选人、职位、时间、格式、主操作
- 状态放在卡片统一位置，不做随机摆放

## 7.7 OfferList
当前问题：
- 顶部缺少汇总指标
- 状态表达不够强

重构方案：
- 顶部 4 卡：待审批、已发送、已接受、已拒绝
- 表格保留核心字段：候选人、职位、部门、薪资、状态、时间
- 主操作保留“创建录用通知”

## 7.8 Inbox
当前问题：
- 通知与任务边界模糊
- 列表项缺少结构化表达

重构方案：
- 明确为“待办流”
- 每项结构：类型标签 + 标题 + 摘要 + 时间 + 主操作
- tab 带计数，支持快速定位

## 7.9 DemandList
当前问题：
- 已接近标准列表页，但缺少顶部指标和节奏整理

重构方案：
- 顶部：需求总数、待审批、进行中、已完成
- 筛选栏：关键词、部门、状态、紧急程度
- 表格固定标准列结构

## 8. 导航与信息架构

### 8.1 全局导航
- TopNav：模块切换
- Sidebar：当前模块子功能
- Breadcrumb：对象路径

### 8.2 上下文对象条
以下页面必须有上下文对象条：
- JobDetail
- CandidateWorkspace
- DemandDetail
- ChannelStaging
- PipelineBoard

### 8.3 操作层级
所有页面操作建议分为 3 层：
- Primary：页面最重要的 1 个动作
- Secondary：辅助动作
- Tertiary：更多菜单或文字链接

## 10. 验收标准

一次合格的重构，应同时满足以下标准：
- 所有核心页面都遵循统一页面骨架
- 列表、工作台、详情、看板、配置页各有固定范式
- 表格、筛选、统计卡、状态标签、空状态全系统统一
- 页面主操作清晰，不超过视觉主路径
- 不同页面之间切换不产生明显的视觉跳跃
- 移动端 / 小屏下基本结构不崩

---

## 11. 最终目标

这次重构不是把页面“变漂亮”，而是让 RecruitOS 变成一个真正可长期演进的产品系统：

- 更清晰
- 更稳重
- 更一致
- 更专业
- 更容易扩展
