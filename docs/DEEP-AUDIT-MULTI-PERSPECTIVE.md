# RecruitOS AI 能力深度审计：四视角交叉分析

**日期**：2026-06-11  
**审计对象**：RecruitOS v6 + recruitos-brain + 8 个触点引擎 + 前端页面  
**审计方法**：工程 / 数据科学 / UX / 业务运营 四个角色独立审视，然后交叉验证

---

## 角色一：工程师视角（关注代码质量、架构合理性、可维护性）

### 我发现的问题

#### 1. BrainDataAggregator 是硬编码模拟数据，不是真正的数据聚合

```java
// 现状：每一个方法都返回写死的 Map
public Map<String, Object> fetchIntentSignals(Long candidateId, Long jobId) {
    m.put("replySpeed", 0.6);  // 硬编码
    m.put("questionsDepth", 0.5);  // 硬编码
    ...
}
```

**根因**：brain 服务没有通过 Feign/HTTP 调用其他微服务（candidate 8085、interview 8086、offer 8087 等）获取真实数据。这意味着所有引擎跑的都是假数据。

**正确做法**：
- brain 应该是 BFF（Backend for Frontend）层，通过 Feign 客户端调用各领域微服务
- 候选人的回复速度应该从 `recruitos-communication` 的消息时间戳计算
- 面试投入度应该从 `recruitos-interview` 的评价文本中做 NLP 分析
- 管道数据应该从 `recruitos-candidate` 的候选人在哪个阶段实时获取

#### 2. 引擎是规则计算器，不是 ML 模型

```java
// IntentPredictionEngine：所有信号权重硬编码
if (speed < 0.5) { score -= 15; }
else if (speed < 1.0) { score += 5; }
else { score += 10; }
```

这不是 AI，这是 if-else 规则引擎。阈值（0.5、1.0）、权重（+5、+10、-15）都是拍脑袋定的。

**正确做法**：
- 意向预测应该是一个分类模型（GBDT 或 LightGBM），从历史 Offer 接受/拒绝数据中训练
- 特征包括：回复速度、提问深度 NLP 向量、薪资差距百分比、历史流失模式
- 模型需要有评估指标（AUC、Precision@K）和持续训练管道

#### 3. LLM 集成方式脆弱

```java
// DemandDiagnosisEngine：手动字符串解析 LLM 返回的 JSON
int jsonStart = raw.indexOf('{');
int jsonEnd = raw.lastIndexOf('}');
String json = raw.substring(jsonStart, jsonEnd + 1);
```

LLM 输出不稳定，JSON 可能被 markdown 包裹、可能缺字段、可能格式错误。这种解析方式在生产环境必崩。

**正确做法**：
- 使用 structured output / function calling（如果 LLM 支持）
- 至少要 JSON Schema 校验 + 字段级 fallback
- 增加重试 + 指数退避
- 每个 LLM 调用记录到 trace，用于后续调试和 prompt 优化

#### 4. 所有计算都是同步的

面试辅助、校准分析、意向预测 —— 全部是同步 HTTP 请求，200ms 超时就崩溃。

**正确做法**：
- 计算密集型任务（如多候选人校准分析）应异步化：提交任务 → 返回 jobId → 轮询结果
- 引入消息队列（RabbitMQ/Kafka）解耦信号采集与计算
- 响应时间敏感的（面试辅助）用 WebSocket 推送而非轮询

#### 5. 前后端分离但脑服务耦合度过高

所有 8 个触点的逻辑全塞在一个 `BrainController` 里，往后每加一个触点就要改这个类。

**正确做法**：
- 按触点拆分为独立的 Controller/Service（如 `IntentController`、`CalibrationController`）
- 共享的底层能力（LLM 调用、特征提取、数据聚合）抽到 common 层

#### 6. 没有持久化层

所有 domain 对象只存在于内存，没有对应的数据库表。进化引擎无法回溯"昨天的意向预测 vs 今天的意向预测"。

**缺失的表**（至少需要）：
- `brain_demand_diagnosis` — 需求诊断历史
- `brain_candidate_intent` — 意向评分历史
- `brain_cycle_prediction` — 周期预测快照
- `brain_calibration_session` — 校准会记录
- `brain_interviewer_profile` — 面试官质量档案
- `brain_talent_density_snapshot` — 人才密度快照
- `brain_ai_decision_log` — AI 决策审计日志

## 角色二：数据科学家视角（关注模型质量、信号有效性、可解释性）

### 我发现的问题

#### 1. 意向预测模型本质上是不可信的

当前意向预测引擎的结果毫无统计学基础：
- 没有训练数据 —— Offer 接受/拒绝是已知结果，但从未被用来训练
- 特征权重是主观赋值的 —— 为什么 `replySpeed < 0.5` 扣 15 分而不是 12 分？
- 信号之间有共线性但没被处理 —— 回复慢的人往往提问也浅，这两个特征叠加会导致双倍扣分
- 没有正负样本 —— 怎么知道当前评分"准"还是"不准"？

**正确的数据科学做法**：
- 用历史 Offer 结果作为 label（接受=1，拒绝=0）
- 特征工程应包括：通信特征（回复间隔分布）、行为特征（面试出席/迟到）、简历特征（跳槽频率）、交互特征（提问类型 NLP）
- 用 SHAP/LIME 解释每个特征的贡献 —— 对 HR 说"意向评 75 分，主要因为候选人回复速度快（+12）和面试投入度高（+18），但薪资预期差（-15）"才是有意义的信息
- 冷启动时用规则兜底，积累 200+ 样本后自动切换到 ML 模型

#### 2. 校准会分析缺少统计严谨性

```java
d.setDisputed(d.getMaxGap() >= 2);  // 分差>=2就是"争议"
```

这太粗糙了。两个面试官对同一维度打 2 分 vs 4 分确实可能是争议，但：
- 可能是面试官 B 挖到了面试官 A 没覆盖的信号，这不一定是"偏差"，可能是"信息不对称"
- 统计上没有做 inter-rater reliability 计算（Cohen's Kappa、ICC）
- 没有区分系统偏差（某面试官长期偏严）vs 随机偏差（单次评分波动）

**正确的做法**：
- 计算每对面试官的 Cohen's Kappa / ICC
- 区分"信号差异"（有人挖到别人没挖到的信息）vs"标准差异"（同样的行为有人打 3 有人打 5）
- 面试官偏差应该用时间序列分析，看评分趋势是否漂移

#### 3. 人才密度评估是假数据

```java
double[] currentLevels = {0.45, 0.70, 0.60, 0.55, 0.40};  // 硬编码
double[] targetLevels = {0.80, 0.75, 0.65, 0.70, 0.60};    // 硬编码
```

**正确的做法**：
- 当前水位应该从 RecruitOS 已有的标签体系（岗位 tags × 团队成员的技能映射）计算
- 目标水位应该从业务目标推导（而不是硬编码 0.80）
- 密度评分应包含：新员工 90 天绩效达标率、内部晋升率、关键岗位离职权重的加权

#### 4. 进化引擎的 L1-L6 信号没有被大脑引擎利用

RecruitOS 的进化引擎已经采集了 L1-L6 信号（面试通过、Offer 接受、试用期结果等），但 brain 服务的所有引擎完全无视这些信号。这是最大的资源浪费 —— 系统最有价值的数据沉淀没有被 AI 决策层使用。

## 角色三：UX 设计师视角（关注用户是否真的会使用、是否嵌入工作流）

### 我发现的问题

#### 1. 页面是"陈列柜"而非"工作工具"

当前所有 AI 页面都是独立路由的展示页面 —— 用户必须专门导航过去看。但真正的 AI 能力应该在用户做事的当下出现。

**反例**：
- 面试官在 `InterviewBoard.vue` 的 Kanban 上拖拽候选人，系统应该在这个页面上弹出一个侧边栏显示 AI 面试辅助，而不是让面试官跳到另一个页面
- HR 在 `PipelineBoard.vue` 看候选人管道，AI 意向预测应该作为候选人的一个标签/徽标直接显示在卡片上，不需要点进专门页面

**正确做法**：
- AI 面试辅助 → 嵌入在 `InterviewBoard.vue` 的右滑面板中
- 意向预测 → 作为 `PipelineBoard.vue` 卡片上的彩色指示点
- 校准会 → 嵌入在 `InterviewEvaluation.vue` 的评价汇总区域
- Offer 策略 → 嵌入在 `OfferList.vue` 的 Offer 创建流程中
- 周期预测 → 嵌入在 `Dashboard.vue` 和岗位工作台
- 面试官质量 → 嵌入在现有 `AnalyticsInterviewer.vue`
- 人才密度 → 嵌入在企业组织架构视图

#### 2. 信息过载，没有渐进式披露

每个页面一次性展示所有 AI 输出 —— 需求诊断页有 5 个 Card、校准会页有 4 个区。用户看到的是数据墙，不是洞察。

**正确的做法**：
- 核心结论优先（一句话摘要 + 置信度）
- 支持细节逐层展开（点击"查看更多"才看到信号细节）
- 区分"必须立刻关注的"（红色）vs"值得了解的"（蓝色）vs"供参考的"（灰色）

#### 3. AI 输出缺少"可操作性"

当前 AI 输出以文本描述为主 —— "建议与 Hiring Manager 沟通业务愿景"。但用户需要的是**可点击的行动**。

**正确做法**：
- AI 输出应该附带 action button：「安排 Hiring Manager 1:1」「发送话术模板」「调整筛选门槛到 X%」
- 点击后直接触发对应操作或弹出确认对话框
- 所有 action 记录到审计日志

#### 4. 面试辅助没有做到"实时"

当前面试辅助是一个"面试前准备页" —— 面试官提前看一眼然后去面试。但真正的 AI 面试辅助应该：
- 面试中实时听/读转录
- 侧边栏显示：「你已经问了 3 个技术问题，Product Sense 维度还没覆盖」
- 面试结束后自动生成结构化记录

目前技术栈不支持这个（没有 WebSocket、没有语音转录），但至少应该设计成面试中可用的轻量交互，而不是面试前的一次性准备。

#### 5. 校准会页面是静态矩阵，不是交互式工具

校准会是一个**活的会议**。AI 应该：
- 实时引导讨论（"我们现在讨论 Product Sense"）
- 让面试官直接在页面上修正评分并记录原因
- 自动记录修正历史
- 在会议结束时生成校准报告

但目前只是一个静态表格展示。

## 角色四：业务运营视角（关注是否真正降本增效、ROI 是否可量化）

### 我发现的问题

#### 1. 没有量化"AI 用了 vs 没用"的效果差

系统部署了 8 个 AI 引擎，但没有任何指标回答：启用这些 AI 能力后：
- 招聘周期缩短了几天？
- Offer 接受率提升了几个百分点？
- 面试评价一致性有没有变好？
- 人才密度是升了还是降了？

**必须建立 A/B 框架**：
- 同一岗位族的招聘，一半启用 AI 辅助，一半不启用
- 对比 90 天后两组岗位的：平均到岗时间、Offer 接受率、入职 90 天绩效
- 如果 AI 组没有统计显著优势，说明这个触点不创造价值，应该关掉或重做

#### 2. "人在回路"没有真正实现

当前系统：AI 给出建议 → 人可以选择采纳或忽略。但这是单向的：
- 人忽略 AI 建议后，AI 不会学习"为什么人被忽略了"
- 人采纳 AI 建议后，AI 不会知道"这个建议最后的业务结果好不好"
- 没有反馈链路

**正确的做法**：
- 每次 AI 建议被忽略时，记录原因（HR 选择"画像不匹配""预算不可行"等选项）
- 每次 AI 建议被采纳后，追踪最终业务结果（招聘到了吗？绩效好吗？）
- 用这些反馈持续优化模型

#### 3. 能力建设没有考虑"冷启动"和"数据飞轮"

当前系统假设所有数据都已经存在。但一个新客户：
- 没有历史 Offer 数据 → 意向预测模型无法训练
- 没有面试评价数据 → 面试官质量无法评估
- 没有入职绩效数据 → 人才密度无法计算

**正确做法**：
- 建立"行业模板"机制 —— 同行业、同规模客户的聚合数据作为冷启动基线
- 数据越丰富，模型越个性化 —— 从行业模板 → 公司模板 → 岗位模板 → 个体模型
- 在数据不足时，明确告知用户「数据不足，当前结果基于行业均值，置信度较低」

#### 4. 没有考虑"AI 出错"的兜底机制

假设 AI 意向预测把"高意向"判为"低意向"→ 系统建议 HR 放弃这个候选人 → 候选人实际上非常想来，只是回复慢。

**必须做**：
- 每个 AI 评分附带置信度，低置信度的建议应该有更柔和的呈现方式
- 关键决策（放弃候选人、拒绝 Offer）不允许 AI 自动执行
- AI 错误应被记录、复盘、用于改进

#### 5. 缺少可配置性

当前所有引擎的阈值、权重、规则都是硬编码在 Java 代码里。一个 HR 经理想调整"面试官偏差告警阈值"需要改代码重新部署。

**正确做法**：
- 提供管理后台，允许租户管理员配置：
  - 意向评分的各等级阈值
  - 面试官偏差告警的宽松/严格程度
  - AI 建议的展示模式（激进/保守）
- 所有配置变更记录审计日志

## 交叉验证：四个角色共同发现的系统性缺陷

### 缺陷 1：引擎与引擎之间完全孤立

每个引擎独立计算自己的结果，但它们本应互相增强：
- 意向预测发现候选人意向低 → 应该触发 Offer 策略引擎建议"提高薪酬或加速流程"
- 校准会发现某面试官系统偏差 → 应该更新面试官质量引擎的偏差档案
- 周期预测发现某阶段瓶颈 → 应该触发需求诊断引擎重新评估"这个岗位的画像是不是定错了"

当前架构没有引擎间通信机制。

### 缺陷 2：数据源是假的，导致所有引擎都是空中楼阁

BrainDataAggregator 返回 mock 数据 → 所有引擎基于假数据计算 → 所有前端展示的都是假数据。这不是演示环境的问题，而是架构设计问题 —— brain 服务没有和真正的数据源建立连接。

### 缺陷 3：重"引擎"轻"嵌入"

我花 80% 的时间写了 8 个引擎的 Java 代码，但真正的用户价值在于"AI 在正确的时机出现在正确的地方"。引擎算力再强，如果不出现在用户的自然工作流中，就等于不存在。

## 方案：从"AI 陈列柜"到"AI 操作系统的四个改造维度

### 改造维度 A：数据层 —— 让引擎吃到真实数据（工程主导）

| 行动 | 优先级 | 工作量 |
|------|--------|--------|
| Brain 服务接入 Feign 客户端，调用各领域微服务获取真实数据 | P0 | 3 天 |
| 建立 `brain_*` 7 张数据库表，所有 AI 结果持久化 | P0 | 2 天 |
| LLM 调用增加 structured output / JSON Schema 校验 | P0 | 1 天 |
| 引入异步计算（CompletableFuture + 消息队列） | P1 | 3 天 |
| 拆解 BrainController 为多个独立 Controller | P1 | 1 天 |

### 改造维度 B：模型层 —— 从规则引擎到统计学习（数据科学主导）

| 行动 | 优先级 | 工作量 |
|------|--------|--------|
| 意向预测：用历史 Offer 数据训练 LightGBM 二分类模型 | P0 | 5 天 |
| 校准会：引入 Cohen's Kappa / ICC 统计指标替代简单分差 | P0 | 2 天 |
| 面试官质量：建立时间序列偏差追踪模型 | P1 | 3 天 |
| 人才密度：从 RecruitOS 标签体系计算真实水位 | P1 | 3 天 |
| 冷启动：建立行业模板聚合机制 | P1 | 3 天 |
| 所有模型增加 SHAP 可解释性输出 | P2 | 2 天 |

### 改造维度 C：嵌入层 —— 让 AI 出现在工作流中（UX 主导）

| 行动 | 优先级 | 工作量 |
|------|--------|--------|
| 意向预测嵌入 Pipeline Kanban 卡片（彩色指示点 + hover tooltip） | P0 | 1 天 |
| 面试辅助嵌入 InterviewBoard 右滑面板 | P0 | 1 天 |
| Offer 策略嵌入 Offer 创建流程 | P0 | 1 天 |
| 校准会改为交互式会议工具（可修改评分、记录修正原因） | P1 | 3 天 |
| 需求诊断改为对话式（多轮追问，不跳过） | P1 | 2 天 |
| 所有 AI 输出增加可点击 action button | P1 | 1 天 |
| Dashboard 增加 AI 价值指标模块 | P2 | 1 天 |

### 改造维度 D：运营层 —— 让系统可度量、可进化（业务运营主导）

| 行动 | 优先级 | 工作量 |
|------|--------|--------|
| 每个 AI 决策记录到 `brain_ai_decision_log` 审计表 | P0 | 1 天 |
| 建立 AI 效果 A/B 框架（启用组 vs 对照组） | P1 | 3 天 |
| 建立"AI 被忽略原因"反馈收集机制 | P1 | 1 天 |
| 管理后台：租户级 AI 配置（阈值、模式、开关） | P1 | 2 天 |
| 冷启动行业模板数据库设计和数据填充 | P2 | 5 天 |

## 诚实的时间线和取舍

如果四个维度全做，按 2 人团队需要约 8-10 周。现实情况下建议分三批：

**第一批（2 周，立刻启动）**：
- 数据层 A1-A3（真实数据接入 + 持久化 + LLM 结构化输出）
- 嵌入层 C1-C4（意向/面试/Offer/需求诊断嵌入工作流）
- 运营层 D1（审计日志）

**第二批（3 周，数据积累后）**：
- 模型层 B1-B2（意向 ML 模型 + 校准统计）
- 运营层 D2-D4（A/B 框架 + 反馈机制 + 配置后台）

**第三批（3 周，持续迭代）**：
- 模型层 B3-B6（面试官时间序列 + 人才密度 + 冷启动 + SHAP）
- 嵌入层 C5-C7（校准会交互式 + 需求诊断对话式 + action button + Dashboard）
- 运营层 D5（行业模板）

## 最关键的一个问题

整个审计下来，最大发现不是技术问题，而是产品哲学问题：

**我们一直在做"AI 告诉我们什么"，而不是"AI 帮我们做什么"。**

意向预测说"这个候选人意向低"—— 然后呢？
校准会说"A 和 B 评分有分歧"—— 然后呢？
需求诊断说"你们缺系统架构能力"—— 然后呢？

每一条 AI 洞察都必须通向一个**可执行的动作**。没有 action 的 insight 只是噪音。
