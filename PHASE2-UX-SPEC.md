# RecruitOS Phase 2 UX 规格（v3 + v3.1）

**状态**：P0 + P1 + P2 已完成  
**对齐决策**：架构 v3（A1–A4、B1–B3、C1–C3）+ 体验 v3.1（U1–U6）

---

## 1. 产品原则

| 原则 | 说明 |
|------|------|
| 一职位一主线 | 在招职位工作台 = 单职位 A（找人）+ B（管人）全链路枢纽 |
| 收件箱 = 动词 | 按「待你确认 / 待面试 / 录用相关 / 招人方式建议 / 待审批」组织 |
| 三層用语 | L1 主界面口语 · L2 展开说明 · L3 高级 JSON（实施顾问） |
| G3 不变 | 招人方式变更须二次确认；运行中平台任务策略锁定 |
| 默认半自动 | 「合适了再联系」+ 发送前请你确认 |

## 3. 信息架构

### 3.1 顶级导航

```
工作台 · 招聘执行 · 招聘规划 · 人才库 · 数据洞察 · 设置
```

菜单改名：

- 策略进化待确认 → **招人方式建议**
- 渠道暂存库 → **跨职位待联系池**
- 沟通 Profile → **对外沟通风格**

### 3.2 在招职位工作台 `/planning/jobs/:id`

**4 个业务 Tab**（非 8 个系统 Tab）：

| Tab | 用户名称 | 子 Segmented | 内容 |
|-----|----------|--------------|------|
| overview | 总览 | — | 双漏斗、编制、单一「下一步」 |
| sourcing | 找人 | 平台招人任务 · 待联系池 | JobSourcing · ChannelStaging(embed) |
| candidates | 管人 | 在招候选人 · 面试与录用 | PipelineKanban · JobHiringSummary |
| rules | 规则 | 职位要求 · 招人方式 · 变更记录 | 要求表 · 跳转 JD/招人方式编辑 · 审计(P1) |

**页头**：职位名 + 状态 + 编制 + 单一主 CTA「下一步：…」

### 3.3 收件箱 `/workspace/inbox`

Tab：`全部` · `待你确认` · `待面试` · `录用相关` · `招人方式建议` · `待审批`

### 3.4 招聘执行（保留，跨职位）

招聘进展 · 候选人 · 面试日历 · 录用通知 · 入职 — 与职位工作台双向 `jobId` 串联。

---

## 4. 关键交互

### 4.1 纳入本职位候选人（A→B 枢纽）

成功后 Toast + 可选跳转：

> 已纳入「{职位}」在招候选人。下一步：在「新人待筛选」中做初筛。

### 4.2 招人方式确认发布

二次弹窗：

> 新的招人方式已生效。只影响**此后新开启**的平台招人任务；正在进行的任务不变。

### 4.3 招人方式建议采纳（B3a）

详情 → 人话摘要 + 依据 → 二次确认 → 发布新版本。

## 6. P0 验收

1. 15 分钟 Demo：需求→职位→招人方式→平台任务→纳入→在招候选人 Kanban  
2. 收件箱可见：待面试、录用相关、招人方式建议、待审批  
3. 主界面无 OpsPack / evolution / CARD_GREET 等工程词  
4. 5 人无培训测试：任务完成率 ≥80%（后续迭代）
