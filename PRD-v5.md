## 一、产品愿景与定位

### 1.1 一句话定义

**Layered on your ATS. Define the job. The system finds, evaluates, converses, interviews, onboards, and learns. You decide.**

RecruitOS 不是要替代客户现有的 ATS（Moka/北森/大易/Workday），而是作为**招聘决策智能层**叠加在上方，接管从 JD 发布到入职的全链路中所有可被 AI 替代的认知劳动。每一次筛选、面试、录用决定都会回流到算法中，驱动岗位模型自我修正。

### 1.2 设计原则

| 原则 | 含义 |
|------|------|
| **算法即活体** | 每个岗位的匹配模型持续从招聘结果中学习，自动进化权重 |
| **人在回路，不在流水线** | 人只做价值判断（筛选、面试），不做机械动作（搜索、打招呼、收简历） |
| **平台无关** | 用企业自有账号模拟真人操作招聘平台，不依赖平台 API 合作 |
| **场景驱动沟通** | 打招呼话术按场景 × 候选人画像 × 历史绩效三级匹配 |
| **租户即王国** | 每个集团拥有完全隔离的数据、模型、配置空间 |

### 1.3 不可替代性

传统 ATS 解决的是"简历来了之后怎么办"，RecruitOS 解决的是"业务提了需求之后怎么办"——从招聘需求审批、岗位创建、去哪里找到对的人、怎么判断、怎么说服他来、到从结果中学习，全链路覆盖。这是一层传统 ATS 不覆盖的能力——需求驱动建岗、主动搜索、两轮面试管理、语义评分、自进化权重、试用期绩效回流验证。

### 1.4 切入策略：先寄生，再共生，后替代

```
Phase 1 — 寄生
  卖"搜索 + 初筛 + 沟通"的增量价值
  简历回收后推回客户现有 ATS
  客户零迁移成本

Phase 2 — 共生
  面试管理、录用审批逐步切到 RecruitOS
  与 ATS 保持双向数据同步
  "RecruitOS 管招聘前端，ATS 管人事后端"

Phase 3 — 替代
  全流程覆盖后，客户可关闭原 ATS 招聘模块
  RecruitOS 成为唯一招聘操作系统
```

## 三、多租户架构与治理

### 3.1 租户隔离

每个租户完全独立：PostgreSQL Schema、Milvus Collection、S3 Bucket、Agent 实例池、进化模型参数。全链路无跨租户数据泄漏。

### 3.2 集团多级组织

```
Tenant (某集团)
├── 集团 HR 中心 (跨公司权限)
├── 子公司 A → 技术部 → 后端/前端/数据团队
├── 子公司 B → 市场部 / 销售部
└── 共享服务中心 (SSC)
```

权限在组织树任意节点授予，向下继承，支持矩阵式权限（一人多角色）。

### 3.3 白标能力

Enterprise 客户可配置：自定义域名 + TLS、品牌视觉（Logo/配色）、邮件模板 + 发件人域名、SSO（飞书/企微/钉钉/OIDC/SAML）。

### 3.4 开放能力

REST API（岗位、候选人、面试、录用、分析数据的 CRUD） + Webhook 事件订阅（评分完成、已打招呼、已回复、简历就绪、面试安排、Offer 发出/接受、入职启动）。

## 四、招聘需求管理

招聘需求是整个招聘流程的起点。**没有经过审批的招聘需求，不允许创建岗位、发布 JD、启动 Agent 搜索。** 这确保了招聘行为始终对齐业务目标和编制预算。

### 4.1 需求提报

用人部门负责人在系统中创建招聘需求，必填字段：

| 字段 | 说明 |
|------|------|
| 需求标题 | 如"后端高级工程师 - 交易系统" |
| 所属部门 | 从组织架构树选择 |
| 招聘人数 | 正整数 |
| 岗位级别 | P6/P7/P8 等，影响薪酬带宽 |
| 期望薪酬范围 | 最低 / 最高月薪，系统自动校验是否在该级别的薪酬带宽内 |
| 紧急程度 | 普通 / 紧急 / 特急，影响 SLA 倒计时 |
| 到岗日期 | 期望候选人入职时间 |
| 需求原因 | 新增编制 / 离职替补 / 业务扩张 / 项目临时用工 |
| 岗位职责描述 | 自由文本，后续作为 JD 解析的输入 |
| 任职要求 | 自由文本，硬性条件 + 加分项 |
| 工作地点 | 支持多地点 |
| 汇报对象 | 直属上级（关联内部用户） |
| 面试官配置 | 指定初面面试官 + 复试面试官 |

### 4.2 审批流

需求提交后进入审批流程，审批链路可按组织层级配置，默认：

```
用人部门负责人提交
    → 部门 HRBP 审核（编制 & 预算校验）
    → HR 负责人审批（跨部门协调 & 优先级排序）
    → [可选] 集团 HRD 审批（总监级及以上岗位）
    → 需求生效 → 自动创建岗位草稿
```

审批支持：通过 / 驳回（附理由） / 转审 / 加签。驳回后可修改重新提交。

### 4.3 需求与岗位的关系

```
招聘需求 (1) ──→ 岗位 (N)
  │                  │
  │  审批通过后       │  JD 解析、标签、权重
  │  自动创建草稿     │  Agent 搜索、候选人关联
  │                  │
  └── 编制 & 预算 ───┘  消耗编制、累计费用
```

- 一个招聘需求可以拆分为多个岗位（如"后端工程师"需求 3 人，可拆为"Java 后端"和"Go 后端"两个岗位）
- 岗位创建时必须关联需求，不允许脱离需求独立建岗
- 岗位关闭时，未消耗完的编制自动归还需求
- 需求可中途关闭，关联的所有岗位同步关闭

### 4.4 需求看板

HR 负责人可查看全部需求的全局视图：

- **需求漏斗**：提报 → 审批中 → 已通过 → 建岗中 → 招聘中 → 已完成 → 已关闭
- **编制消耗率**：各需求已录用人数 / 需求人数
- **SLA 告警**：超期未建岗、超期未到岗自动提醒
- **需求来源统计**：按部门、紧急程度、需求原因聚合

## 五、JD 智能解析引擎

### 5.1 输入与输出

招聘需求审批通过后，系统自动创建岗位草稿。HR 在草稿基础上补充或调整 JD 文本，触发解析流程：

**需求描述（自动带入）** → HR 微调 → **触发 JD 解析** → 语义分块 → 实体抽取 → 标签生成 → 权重分配 → 岗位画像。

需求中的岗位职责和任职要求作为 JD 解析的主要输入源，HR 可在此基础上补充细节，无需从零编写。

**五类标签**：硬性标签（必须）、软性标签（加分）、行业标签（领域）、否定标签（排除）、语义向量（768 维 embedding）。

### 5.2 三元权重

每个标签带三个独立权重：

```json
{
  "tag": "Kubernetes 生产级运维",
  "match_weight": 0.9,
  "search_weight": 0.8,
  "decision_weight": 0.7,
  "locked": false,
  "correlations": {"Docker": 0.85, "DevOps": 0.72},
  "evolution_history": []
}
```

- **match_weight**：简历匹配计分权重
- **search_weight**：平台搜索关键词权重
- **decision_weight**：最终排序推荐权重

初始值由 LLM 根据 JD 措辞强度推断（"必须具备"→1.0，"加分项"→0.4）。

### 5.3 冷启动

新租户无历史数据时，加载同行业预训练权重模板。模板由平台脱敏聚合所有租户数据训练，覆盖互联网、金融、制造、医疗等 6 个行业。

## 六、算法进化引擎

### 6.1 六级进化信号

```
L1 录用/拒Offer         置信度 1.0  →  全标签权重调整
L2 面试通过/不通过       置信度 0.8  →  相关标签权重调整
L3 简历筛选通过/淘汰     置信度 0.6  →  标签阈值调整
L4 候选人回复率          置信度 0.4  →  沟通策略调整
L5 打招呼→简历转化率     置信度 0.3  →  搜索标签 + 话术调整
L6 入职后试用期绩效      置信度 0.9  →  岗位模型全局修正（来自 HRone 回流）
```

### 6.2 协同学进化

每个标签不是独立调整的。系统维护标签协方差矩阵，当一个标签权重变化时，关联标签按协方差比例联动：

```
W_tag_j_new = W_tag_j_old + α × signal × confidence × delta_i × Σ_ij
```

这解决了"面试官因 Docker 录用某人但只有 K8s 权重上升"的失真问题。协方差矩阵初始来自行业预训练模板，随租户数据积累逐步个性化。

### 6.3 进化节奏控制

| 阶段 | 样本数 | 学习率 α | 单次权重上限 |
|------|--------|----------|-------------|
| 快速学习期 | 1-20 | 0.15 | ±0.18 |
| 收敛期 | 21-50 | 0.08 | ±0.12 |
| 稳定期 | 51+ | 0.02 | ±0.06 |

### 6.4 模型健康度监控

系统持续追踪四个指标，任一跌破阈值即告警并建议回滚上一个健康快照：

| 指标 | 告警阈值 |
|------|----------|
| 预测准确率（Top10 评分 vs 面试通过率的 30 天移动平均） | < 50% |
| 权重稳定性（标签权重 30 天方差） | > 0.15 |
| 面试官一致性（不同面试官评分的 Krippendorff's alpha） | < 0.6 |
| 过拟合检测（历史数据 vs 新数据的评分偏差） | > 30% |

### 6.5 A/B 验证

权重或策略变更进入待确认队列前，可先走 A/B 验证：80% 流量走建议组，20% 走当前生效组。2 周或累计 20 个信号后对比回复率、简历转化率、面试通过率。**A/B 胜出仅作为建议可信度标注，不自动切换**——仍须 HR 在「策略进化待确认」中点击确认后才正式采用。

### 6.6 进化锁定

三种锁定粒度：岗位级（整个模型冻结）、标签级（特定标签权重锁定）、维度级（锁定 match/search/decision 某一维）。锁定期间调度器继续采集信号，但不生成可应用的建议。

### 6.7 策略进化对象（v5.1）

进化引擎不仅调整标签三维权重，还覆盖渠道运营三类策略：

| 进化对象 | 对应配置 | 主要信号 |
|----------|----------|----------|
| **搜索关键词** | 标签 `search_weight`、Agent 搜索词列表 | L5 打招呼→回复/简历；L2/L1 录用者标签 |
| **筛选策略** | 两阶段规则、`passThreshold`、`match_weight` | L3 初筛通过/淘汰；L2 面试结果反推 |
| **话术策略** | 场景话术排名、默认模板、复聊间隔/次数 | L4 回复率；L5 简历转化；L1 Offer 接受 |

每次进化输出 **渠道运营包版本**（`SourcingOpsPack vN`），与 `job_weight_snapshot vN` 一一关联。

### 6.8 信号自动采集（v5.1）

各业务模块在事件完成时**异步**发射信号，禁止依赖 HR 手工提交或页面按钮。

| 级别 | 业务事件 | 发射模块 |
|------|----------|----------|
| L3 | 初筛通过 / 淘汰 | `recruitos-agent`（`campaign_candidate_trace` 状态变更） |
| L4 | 候选人回复 / 沉默超时 | `recruitos-communication` |
| L5 | 打招呼发出 / 收简历 | `recruitos-agent` + `recruitos-communication` |
| L2 | 面试通过 / 不通过 | `recruitos-interview`（结构化评价提交） |
| L1 | Offer 接受 / 拒绝 | `recruitos-offer` |
| L6 | 试用期通过 / 淘汰 | `recruitos-onboard` / HRone 回流 |

统一接口：`POST /api/evolution/signals/emit`（内部 SDK `EvolutionSignalEmitter`）。信号字段对齐 `evolution_signal` 表：`signal_level`、`confidence`、`candidate_id`、`tag_adjustments` JSON。

### 6.9 进化建议与 HR 确认门禁（v5.1，强制）

**核心原则**：进化引擎只负责**计算建议**，任何策略变更**不得自动写入线上配置**。全部变更须 HR 点确认后才生效。

```
业务事件 → 信号队列 → 调度器计算 → evolution_proposal（PENDING）
    → HR 待确认队列 → 确认应用 / 驳回 / 过期
    → 写入 SourcingOpsPack vN+1 + WeightSnapshot vN+1 → 仅影响新启动的 Campaign
```

**调度触发**：

| 触发 | 条件 | 动作 |
|------|------|------|
| 批次进化 | 同岗位未处理信号 ≥ 20，或每日 02:00 | 生成合并建议包 |
| 轻量建议 | 同岗位 L4/L5 信号 ≥ 5 | 可生成话术类建议（仍须 HR 确认） |

**`evolution_proposal` 字段**：

| 字段 | 说明 |
|------|------|
| `job_id` | 关联岗位 |
| `proposal_type` | `OPS_PACK` / `WEIGHT_ONLY` / `SCRIPT_ONLY` |
| `status` | `PENDING` / `APPLIED` / `REJECTED` / `EXPIRED` |
| `diff_json` | 变更前后 diff（人可读摘要 + 机器可读明细） |
| `evidence_json` | 信号统计、时间窗、转化率对比 |
| `ab_result_json` | 可选，A/B 胜出证据 |
| `expires_at` | 默认 7 天未处理自动 `EXPIRED` |

**HR 工作台「策略进化待确认」**：每条卡片展示变更类型、摘要、依据、预期影响；支持 **整包确认**（同次进化合并为 1 条）或 **整包驳回**（须填写原因）。确认后通知岗位负责人；驳回原因写入进化审计日志供后续学习。

**运行中 Campaign 不热更新**：已启动的渠道招聘任务继续使用确认生效时的运营包版本，避免执行中策略突变。

### 6.10 健康告警与回滚建议（v5.1）

模型健康度监控（§6.4）跌破阈值时，**不自动回滚**。系统生成 **`ROLLBACK` 类型进化建议** 进入待确认队列，附告警详情与建议恢复的快照版本。HR 确认后执行回滚并归档当前版本。

## 七、多平台 Agent 集群

### 7.1 设计原则

每个租户独立 Agent 实例池，每个平台专属 Agent 进程。模拟真实人类操作——自然间隔、浏览行为、鼠标轨迹。使用企业自有账号密码（KMS 加密存储），不依赖平台 API。

### 7.2 完整工作流

```
搜索 → 快速评分 → Top100 排序 → Top10 打招呼 → AI 沟通索要简历
                                                    │
                                              简历下载 → 解析入库 → 通知面试官筛选
                                                    │
                                         有人拒绝 → Top11 自动补位
                                         3天沉默 → 激活话术
```

### 7.3 UI 变更容错：三明治校验

这是 Agent 可靠性的核心设计。每次操作前执行三层校验：

1. **页面指纹**：计算目标页面 DOM 结构哈希，不匹配则暂停告警
2. **语义定位**：不依赖 CSS 选择器，用视觉+语义定位目标元素
3. **执行验证**：动作后截图对比预期，异常则重试 2 次

Agent 完全不可用时，降级为**半自动模式**：系统生成搜索 URL + 操作指引，HR 手动执行后结果回传。业务永不停。

### 7.4 行为随机化防检测

| 行为维度 | 随机化策略 |
|----------|-----------|
| 操作间隔 | Gaussian 分布：搜索 15-45s，消息 60-180s |
| 浏览轨迹 | 随机上下滚动，随机停留 3-8 秒 |
| 活跃时段 | 模拟 9:00-18:00 工作时段，夜间休眠 |
| 日上限 | 打招呼 ≤ 50 次/天，搜索 ≤ 10 批次/天 |

### 7.5 账号健康度评分

监控验证码频率（权重 0.35）、功能受限警告（0.30）、消息成功率（0.20）、登录异常（0.15）。健康度 < 60 自动降频，< 30 暂停并通知 HR。

### 7.6 合规模式选项

提供三种模式供客户选择：全自动 / 半自动（预览确认后发送） / 仅搜索（手动打招呼）。合同明确平台账号由客户提供，自动化风险客户评估承担。

### 7.7 平台覆盖

Boss 直聘 · 猎聘 · 脉脉 · LinkedIn · GitHub/Stack Overflow（被动候选人扫描）

## 八、AI 沟通引擎

### 8.1 场景-简历-话术三级策略

沟通由三个维度决定发送什么：**什么场景**（初次打招呼/追问/索要简历/面试邀请/Offer/沉默激活）× **候选人画像**（技术型/管理型/应届生/竞品员工）× **话术绩效**（历史回复率、简历转化率）。

### 8.2 条件话术决策树

话术不是简单变量替换，而是条件决策树：

```
IF 候选人当前公司 ∈ 竞品列表 → 竞品定向话术
ELIF 候选人 3 个月内活跃 → 快速转化话术
ELIF 候选人级别 = 高管 → 高管专属话术
ELSE → 标准初次沟通话术
```

每条话术追踪全生命周期绩效（发送→回复→简历→面试→录用→留存），自动排名推荐高绩效话术，支持 A/B 测试。

### 8.3 安全审查三道闸门

AI 生成的消息在发送前必须通过：

1. **薪资承诺检测**：AI 给出的数字超出岗位预算 → 拦截
2. **敏感话题检测**：政治、性别、年龄、地域歧视等 → 拦截告警
3. **Prompt Injection 检测**：候选人回复包含操控 AI 的指令 → 拦截 + 升级人工

当检测到投诉情绪、法律威胁词汇，对话自动升级给人工 HR。全量对话记录不可删除，支持合规审查。

### 8.4 六模块沟通配置与租户继承（v5.1）

在 §8.2 决策树之上，增加可继承的 **CommunicationProfile**（租户默认 + 岗位差异覆盖）：

| 模块 | 说明 |
|------|------|
| AI 人设 | 沟通风格、称谓、语气 |
| 公司背景 | 公司介绍，用于首次触达 |
| 岗位信息 | 从 JD 自动填充，只读 |
| 沟通逻辑 | 主动提问、分支话术 |
| 主动索要触发 | 有聊天记录 / 意向信号 / 第 N 轮 → 索要简历或联系方式 |
| 注意事项 | 薪资保密、禁过度承诺、敏感词规避 |

新岗位自动继承租户默认 Profile；岗位仅覆盖差异模块。运行时由决策树选择场景实例话术；话术绩效（回复率、简历转化）发射 L4/L5 信号，驱动 §6.7 话术类进化建议（仍须 HR 确认）。

### 8.5 复聊策略 RechatPolicy（v5.1）

| 配置 | 默认 |
|------|------|
| 触发 | 已读不回 ≥ 24h；聊天未结束且非已获简历/已拒绝 |
| 最大次数 | 2 |
| 间隔 | 48h |
| 工作时段 | 继承 Agent 活跃时段 9:00–18:00 |

候选人明确拒绝后停止复聊；每次复聊留痕。复聊成功率进入话术进化证据。话术变更走 §6.9 HR 确认门禁。

## 十、面试与录用

面试固定为**两轮制**：初面 + 复试。面试官在招聘需求提报时指定，确保候选人体验一致且流程可控。

### 10.1 初面

- **面试官**：用人部门直属负责人或团队技术骨干（在招聘需求中配置）
- **考察重点**：专业技能深度、岗位匹配度、技术实操能力
- **时长**：默认 60 分钟，可按岗位调整
- **形式**：线上（腾讯会议/飞书/Zoom）或线下，系统自动协调面试官日历与候选人偏好，推荐 3 个候选时间
- **技术岗位**：自动创建在线编程环境
- **面试录制**：自动录制 + AI 生成摘要（经候选人同意）
- **初面评价**：面试官提交结构化评价，系统判定"通过/待定/淘汰"。通过后自动触发复试安排

### 10.2 复试

- **面试官**：部门负责人、HR 负责人或跨部门面试官（在招聘需求中配置）
- **考察重点**：综合能力、文化匹配度、发展潜力、薪酬预期
- **时长**：默认 45 分钟
- **形式**：与初面一致，系统自动创建会议链接
- **复试评价**：面试官提交结构化评价，最终决策"录用/待定/淘汰"
- **待定处理**：可追加一轮加面（系统记录为"复试补充"），但不改变两轮制结构

### 10.3 结构化评价

初面和复试均使用统一的结构化评价模板，强制填写，作为进化引擎 L2 信号源：

```json
{
  "round": "initial",
  "decision": "pass",
  "overall_score": 85,
  "dimensions": [
    {"name": "技术深度", "score": 90, "comment": "K8s 源码级理解"},
    {"name": "系统设计", "score": 78, "comment": "高并发经验弱"}
  ],
  "evolution_feedback": {
    "confirmed_tags": ["Kubernetes", "Go"],
    "weakened_tags": ["高并发系统"],
    "suggested_new_tags": ["技术管理潜力"]
  }
}
```

初面和复试的评价维度可分别配置：初面侧重技术能力维度，复试侧重综合素养维度。

### 10.4 面试流程状态机

```
初面待安排 → 初面已安排 → 初面进行中 → 初面已完成
    ↓ 淘汰      ↓ 淘汰       ↓ 淘汰       ↓ 初面通过
  流程结束     流程结束     流程结束    → 复试待安排
                                          ↓
                                  复试已安排 → 复试进行中 → 复试已完成
                                      ↓ 淘汰       ↓ 淘汰       ↓ 录用决策
                                    流程结束     流程结束    → 发起 Offer
```

### 10.5 Offer 与背调

Offer 审批流（面试官→HR→部门负责人→薪酬委员会，可配置）、Offer 模板按级别/地区预设、电子签对接（法大大/上上签）。背调对接太和鼎信 + 学信网。

## 十五、安全架构

### 15.1 防白嫖：许可与激活

**试用期防护**：注册必须企业邮箱 + 营业执照 OCR + 企业主体唯一性校验。同一社会信用代码不可重复试用。试用期禁止数据批量导出。

**许可验证**：每次 API 请求在 Gateway 层验证租户许可（RS256 签名），检查过期时间、状态、功能开关、用量限额。过期 7 天内进入宽限期（可用但提醒续费），超 7 天返回 402。

**用量防篡改**：所有计数器在服务端 Redis 原子操作，双写 TimescaleDB + 不可变日志（S3 WORM），每日自动三方对账，不一致即时告警。

### 15.2 认证与会话

- 密码 ≥ 12 位 + MFA（TOTP 或 WebAuthn）
- SSO：SAML/OIDC 对接飞书/企微/钉钉
- JWT Access Token 1 小时 + Refresh Token 24 小时（HTTP-only, SameSite=Strict）
- Refresh Token 旋转（使用后旧 Token 立即失效）
- 同用户最多 5 个活跃会话，管理员可远程踢出

### 15.3 多租户数据隔离

五层隔离：应用层 SQL 注入 tenant_id → PostgreSQL RLS 策略兜底 → Milvus Partition Key → S3 Bucket Policy → 日志脱敏。跨租户查询在代码层被拦截，数据库层兜底，任何不带 tenant_id 的查询触发告警。

### 15.4 API 与网络安全

- 四层限流：IP（1000/min）→ 租户（按 plan 差异化）→ 端点（登录 5/min 防爆破）→ 用户
- WAF：OWASP CRS 3.x + SQL 注入/XSS/路径遍历检测
- 请求签名防重放：HMAC-SHA256(timestamp + nonce + body_hash)
- 安全头：CSP, HSTS, X-Frame-Options: DENY, Referrer-Policy
- 网络：Cloudflare DDoS → LB → API GW（多实例）→ VPC 内网微服务 → 数据库公网不可达

### 15.5 Agent 凭证安全

平台账号密码：前端 RSA-OAEP 加密传输 → 后端 KMS 解密后重新用租户专属 AES-256-GCM 加密存储。明文仅在 Agent 进程内存短暂存在，不落盘。数据库字段是密文，KMS 密钥在 HSM 中。每次 KMS 解密操作独立审计，异常频率自动告警。

### 15.6 代码与供应链

SAST（Semgrep/CodeQL）+ 密钥扫描（truffleHog）+ 依赖扫描（Snyk）在 CI 自动执行。容器镜像 Trivy 扫描 + Cosign 签名 + 部署时验签。

### 15.7 安全运营

- 异常检测：暴力破解 → IP 自动封禁 · 用量突增 → 限流 · 跨地域登录 → 强制 MFA
- 告警升级：自动处理 → 通知客户 → PagerDuty 安全团队 → CTO + 法务
- 渗透测试：CI 自动扫描 + 季度内部渗透 + 半年第三方渗透
- 认证目标：Phase 2 完成等保三级 + SOC2 Type I，Phase 3 完成 SOC2 Type II + ISO27001
